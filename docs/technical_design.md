# 技术设计说明

## 1. 问题定义

直接把整本 PDF 丢给大模型会带来几个问题：

1. 上下文太大，成本高，稳定性差。
2. 模型容易脱离书本内容自由发挥。
3. 学习进度、历史课程和复习记录难以保存。
4. 编程类书和文科类书需要不同学习方法，不能只用同一种总结模板。

因此，本项目把“整本书输入”改造成“可检索、可推进、可保存”的课程资料库。

## 2. 系统目标

- 让 AI 严格围绕一本书或一段书籍内容教学。
- 学习过程按 lesson 推进，而不是一次性总结整本书。
- 每次只取当前课程相关页码和检索片段进入上下文。
- 保存学习进程、课程输出、随时问答和模型配置。
- 根据书籍类型使用不同教学策略。
- 模型不可用时返回错误，不生成本地假内容。

## 3. 核心流程

```text
PDF path
  -> ingest
  -> books / sections / chunks
  -> create session
  -> page-window plan
  -> retrieve current lesson content
  -> build learning-mode prompt
  -> call LLM
  -> save lesson output
  -> show in web UI
```

## 4. Ingest

输入：PDF 文件路径。

输出：

- `books`：书籍元信息。
- `sections`：目录或估算章节。
- `chunks`：带页码和章节信息的文本片段。

流程：

1. 使用 `PyMuPDF` 读取 PDF。
2. 优先读取 PDF 目录。
3. 如果有目录，按目录生成章节范围。
4. 如果没有目录，按页码窗口生成估算章节。
5. 将章节正文拆成 chunk，并保留页码范围。

## 5. Planner

Planner 负责生成学习计划。

当前策略是页码窗口：

- 不再提前固定整本书所有课程。
- 每节课覆盖连续页码。
- 用户可以在学习过程中调整未来每课页数。
- 模型可以建议未来阅读页数，后端会重建后续课程。

这样可以避免一开始就把课程数量写死，也能适应“某些章节内容很密，需要少读几页”的情况。

## 6. Retriever

Retriever 负责为当前 lesson 找到相关书籍片段。

策略：

1. 优先限制在当前 lesson 对应页码和章节中。
2. 使用 SQLite FTS5 做全文检索。
3. 使用中文分词和词法重排提高相关性。
4. 返回少量高相关 chunk，并带上页码。

这样可以减少上下文成本，同时让模型回答时保持书本依据。

## 7. Learning Modes

学习模式 prompt 位于 `book_agent/learning_modes/`。

`programming` 模式强调：

- 核心概念
- 通俗解释
- 代码拆解
- 运行结果预测
- 动手修改
- 练习题
- 常见错误
- 小项目实践

`humanities` 模式强调：

- 本章问题意识
- 作者核心观点
- 论证结构
- 案例与论据
- 现实联系
- 批判性思考
- 复述训练

两个模式都要求模型不要轻易向用户抛问题。需要提问时，应优先给出任务、练习或表达模板，让用户可以直接继续学习。

## 8. Tutor

Tutor 是教学编排层，主要职责：

- 读取当前 session、lesson 和学习模式。
- 检索当前 lesson 相关 chunk。
- 拼接系统 prompt、课程信息、书籍片段和用户问题。
- 调用 OpenAI 兼容大模型接口。
- 保存主线课程输出。
- 区分主线课程和随时问答。

如果大模型调用失败，Tutor 返回错误，不保存课程输出。

## 9. 数据模型

主要表：

- `books`：书籍元信息。
- `sections`：章节树。
- `chunks`：可检索文本片段。
- `plans`：学习计划。
- `sessions`：学习进程。
- `messages`：对话消息。
- `lesson_outputs`：每课保存的主线学习内容。
- `app_settings`：网页端保存的模型配置。

## 10. 前端设计

前端位于 `book_agent/web/`，使用原生 HTML、CSS 和 JavaScript。

主要区域：

- 学习进程列表。
- 新建学习进程。
- 模型配置。
- 当前课程内容。
- 模块分页。
- 上一课和下一课。
- 随时提问区。

主线课程和问答区分离，避免用户提问后覆盖原来的学习内容。

## 11. 为什么暂时不用向量数据库

MVP 阶段使用 SQLite FTS5，而不是 FAISS、ChromaDB 等向量数据库，原因是：

- 依赖更少，安装更简单。
- 更适合先验证产品流程。
- PDF 页码、章节、课程进度和保存逻辑比 embedding 检索更优先。
- 后续可以在 `retrieval.py` 内替换检索实现，不影响上层 API。

## 12. 后续演进

1. 浏览器直接上传 PDF。
2. 大模型流式输出。
3. 每个学习进程独立模型配置。
4. Markdown 渲染和代码高亮。
5. 课程地图和复习状态。
6. 笔记、练习完成记录和间隔复习。
7. 更细的图表、公式、代码块提取。
